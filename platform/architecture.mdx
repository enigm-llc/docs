Enigm is designed as a privacy-first communications ecosystem, not as a single-purpose messaging application surrounded by unrelated services. The architecture is built around a simple principle: protected communications should remain confidential while supporting device trust, administrative governance, dedicated private environments, privacy-oriented connectivity, emergency workflows, and security visibility.

Enigm is the private messaging product at the center of the user experience. Supporting products provide web administration, dedicated private messaging environments, secure operating-system hardening, emergency alerting, private connectivity, payment privacy, and defensive intelligence. Each product is documented as a separate trust domain so that security reviewers can evaluate what the component does, what it does not do, and how it interacts with protected content.

## Overview

Enigm provides the primary account, communication, key-management, and multi-device experience. Enigm App also integrates optional VPN Service usage and Proxy Network traffic separation where enabled. The platform around it exists to support policy, visibility, secure networking, Device Trust lifecycle, threat intelligence, and controlled release without turning those supporting systems into plaintext access paths.

Enigm OS is an optional dedicated secure device layer. When deployed, it contributes Trust Security Center posture, device-level policy enforcement, Controlled Device Management, OTA Architecture, Remote Attestation, and update verification. It is not the center of the platform; it is an optional hardening layer for deployments that require dedicated device control.

The primary products are Enigm, Enigm Command, Enigm Server, Enigm OS, Enigm Key, and Enigm eSIM. Supporting components such as VPN Service, Proxy Network, Tor Gateway, Payment Privacy, Threat Intelligence Platform, and Enyra extend those product surfaces without changing the product taxonomy.

Enigm-controlled servers and services are operated under Enigm's Swiss subsidiary and Swiss legal governance. Region selection for Enigm Server controls product deployment region categories, but it does not change the Enigm legal governance model or the separation between administration and protected content confidentiality. This governance statement does not imply that Enigm operates third-party carrier infrastructure used for Enigm eSIM carrier-layer connectivity.

The architecture is intentionally layered. Account Trust, Device Trust, administrative authorization, network privacy, OTA Eligibility, Remote Attestation, and content confidentiality are separate concepts. Keeping those concepts separate reduces the risk that one administrative, commercial, network, or device-management workflow becomes an unintended authority over private communications.

## Architecture objectives

- Keep Enigm as the primary user-facing private messaging product.
- Preserve content confidentiality as the core product boundary.
- Provide policy-aware supporting services without making them implicit trust roots.
- Support optional Enigm OS posture for dedicated secure-device deployments.
- Use Privacy-Preserving Device Handles for secure device management and audit correlation.
- Protect OTA Architecture with release lifecycle controls, Remote Attestation as an OTA Eligibility signal, client verification, controlled rollout infrastructure, and Hardware-Backed Signing.
- Support optional network privacy and traffic separation through VPN Service and Proxy Network within Enigm App usage.
- Support dedicated private messaging environments through Enigm Server.
- Support selected web-surface access through Tor Gateway under Enigm Command.
- Support mobile connectivity, emergency alerting, and payment privacy through Enigm eSIM, Enigm Key, and Payment Privacy.
- Process security signals through Threat Intelligence Platform and Enyra.
- Govern administrative workflows through Enigm Command.
- Encrypt metadata according to the applicable product, storage, and security domain while preserving only the operational identifiers required for routing, authentication, availability, security, and compliance.

## Core components

### Enigm

Enigm is the private messaging product and primary user-facing security surface. It is delivered through the app and participates in account lifecycle, key management, secure messaging, secure calls, Active Defense, multi-device workflows, optional VPN Service usage, and integrated Proxy Network usage where enabled. It consumes policy and device state rather than assuming that account authentication alone is sufficient for protected operations.

### Enigm OS

Enigm OS is an optional dedicated secure device layer. It provides security architecture, setup controls, launcher policy, network policy, privacy mode, Controlled Device Management, and OTA verification behavior for deployments that require OS-level hardening.

### Trust Security Center

Trust Security Center is the device posture interface. It represents security state for users and administrators, including supported network-policy, privacy, device-management, and update-related signals.

### Enigm Command

Enigm Command is the web control panel product for individual users, organizations, and enterprise administrators. It supports policy assignment, account lifecycle, full account deletion, platform data deletion, session review and closure, connected-device visibility, unauthorized device removal, device revocation, Enigm OS managed-device mode when enabled by the user, Enigm Server purchase and creation, server ID join request management, Enigm eSIM purchase and lifecycle review, Enigm Key lifecycle visibility, Enyra Product Assistant, Tor Gateway access to selected web surfaces, rollout review, audit review, and security posture workflows. Enigm Command actions should be authenticated, authorized, scoped, and auditable.

### Enigm Server

Enigm Server provides dedicated private messaging environments for individual users, teams, and organizations. It supports dedicated administrative boundaries, server ID join request approval, server membership control, server-scoped encrypted content lifecycle and availability controls, user-selected geographic deployment region, and reduced exposure without replacing Enigm App end-to-end encryption or Device Trust.

Enigm Server administration and content confidentiality are separate trust domains. Administrative controls operate on encrypted content objects and lifecycle state; they do not grant access to message plaintext, attachment plaintext, user communications, private key material, or cryptographic authority.

The server ID is used for join requests. It is not an access credential and does not bypass administrator approval, Account Trust, Device Trust, protected key material, or server policy.

### OTA Architecture

OTA Architecture manages update security from preparation through release lifecycle, signing, controlled rollout, client verification, and post-release review. Remote Attestation evaluates device state before OTA Eligibility or other sensitive actions that require device-integrity evidence.

### Hardware-Backed Signing

Hardware-Backed Signing protects release artifacts, policy bundles, configuration bundles, and security-sensitive device-facing materials. Public documentation describes signing at a trust-boundary level and avoids deployable signing details.

### VPN Service and Proxy Network

VPN Service and Proxy Network are associated with Enigm App and platform traffic flows where enabled. VPN Service provides optional network privacy and transport protection. Proxy Network provides traffic separation and reduces direct exposure between app clients and platform services. Neither component is Enigm Server, and neither replaces end-to-end encryption, Device Trust, or secure messaging.

### Tor Gateway

Tor Gateway provides a privacy-oriented access path for selected public web surfaces. It is not Enigm Server, not an Enigm Command administrative surface, not a general-purpose infrastructure access path, and not intended for internal APIs, development systems, operational tooling, or sensitive infrastructure management.

### Enigm Key

Enigm Key is a privacy-oriented physical emergency connectivity device. It supports account-associated emergency alerting, selected-contact notification, and event-bound location sharing while remaining dormant during normal non-emergency operation.

Initial linking and emergency contact configuration are Enigm App workflows. Enigm Command provides device administration, lifecycle visibility, and revocation workflows without becoming a routine location-tracking or emergency-contact configuration surface.

### Enigm eSIM

Enigm eSIM provides data-only, privacy-oriented mobile connectivity across supported coverage areas. It is purchased through Enigm Command, linked to the user's Enigm account for lifecycle management, and can be unlinked, deleted, or retired by the user.

The Enigm-side eSIM purchase and lifecycle-management workflow is identity-minimizing and does not collect KYC verification, email address, phone number, or identity document. Enigm is not a mobile network operator, mobile virtual network operator, telecommunications carrier, radio access network operator, or direct issuer of the underlying carrier connectivity. Carrier-layer mobile data service is operated by an independent telecommunications infrastructure provider.

Enigm eSIM is separate from Enigm App cryptography, secure messaging, VPN Service, Proxy Network, and Device Trust. It can be combined with VPN Service and Proxy Network, but it does not replace either control.

### Payment Privacy

Payment Privacy describes supported commercial workflows that are designed to reduce unnecessary identity linkage while preserving security, accounting, legal, and operational boundaries. Enigm payments are managed through Enigm Command during enrollment and from within Enigm Command after enrollment. Supported methods include cryptocurrency payments, credit card payments, and Code Coin payment-code workflows.

Payment workflows are separate from Enigm App cryptography, secure messaging, secure calls, Device Trust, Enigm Server administration, and protected content access. Standard payment enrollment does not require email address, phone number, or identity document collection; the purchase country is selected by the user.

### Threat Intelligence Platform and Enyra

Threat Intelligence Platform processes security signals for detection, risk modeling, and blocking architecture. Enyra is a user-facing intelligence capability in the Enigm ecosystem that contributes evaluated outcomes to authorized user review or defensive workflows. These systems should support data minimization, privacy-preserving identifiers, and category-level risk explanations.

## Boundary model

The platform separates app, optional OS, network, intelligence, administrative, and release-governance responsibilities.

### App boundary

The app boundary covers account state, secure messaging, secure calls, key-management flows, and multi-device state.

### Device boundary

The device boundary covers optional Enigm OS policy, Trust Security Center posture, privacy mode, network policy, Controlled Device Management, and OTA Eligibility.

### Network boundary

The network boundary covers VPN Service, Proxy Network, Enigm eSIM connectivity, Enigm Key connectivity, network-policy decisions, and related audit outcomes.

The network boundary is designed to reduce direct exposure between user devices and platform services. Limited routing, request-handling, authentication, and security identifiers remain necessary to operate the platform. These identifiers should be minimized, encrypted according to the applicable product and storage domain, and separated from message plaintext, secure call content, media, attachments, user conversations, and private key material.

### Intelligence boundary

The intelligence boundary covers Threat Intelligence Platform ingestion, normalization, Enyra evaluation, risk categorization, and blocking architecture.

### Administrative boundary

The administrative boundary covers Enigm Command, Enigm Server management, Tor Gateway access to selected web surfaces, policy assignment, device lifecycle actions, release review, and audit review.

### Release boundary

The release boundary covers OTA Architecture, Hardware-Backed Signing, Remote Attestation as an OTA Eligibility signal, client verification, controlled rollout, and rollback planning.

## Reference flow

1. A user authenticates in Enigm.
2. Enigm App evaluates account, session, and multi-device state.
3. Optional Enigm OS contributes device posture through Trust Security Center and policy state where deployed.
4. A Privacy-Preserving Device Handle supports device lifecycle and audit correlation.
5. Enigm Command applies authorized policy for user, organization, enterprise, and administrative workflows.
6. VPN Service and Proxy Network apply optional app-level network privacy and traffic-separation controls where enabled.
7. Enigm Command manages Enigm Server lifecycle, server ID join requests, server membership, Tor Gateway access to selected web surfaces, and administrative policy.
8. Enigm Server provides a dedicated private messaging environment for approved Enigm users.
9. Enigm eSIM, Enigm Key, and Payment Privacy apply connectivity, emergency alerting, or commercial policy where enabled.
10. Threat Intelligence Platform and Enyra process security signals for risk or blocking outcomes.
11. OTA Architecture verifies release metadata and artifacts through client verification, Remote Attestation as an OTA Eligibility signal, and Hardware-Backed Signing.
12. Controlled rollout and rollback workflows govern broad exposure of security-sensitive changes.

## Security considerations

- Enigm account authentication should not imply Device Trust or administrative authorization.
- VPN Service and Proxy Network should enforce network privacy and traffic-separation policy without becoming broad data-collection points.
- Enigm Server administration should manage server membership and encrypted content lifecycle without becoming a plaintext, attachment, user communication, or private key access surface.
- Enigm Command should be separated from routine user workflows.
- Optional Enigm OS posture should harden supported deployments without making app security dependent on OS-only assumptions.
- OTA Architecture should reject artifacts that fail verification.

## Privacy considerations

The platform is privacy-oriented by design. It should minimize collection, separate protected content from operational metadata by trust domain, and use Privacy-Preserving Device Handles for device lifecycle and audit correlation.

Network-policy metadata, intelligence signals, device-management records, and audit records should be classified, encrypted, minimized, and retained only for defined product, security, legal, or operational purposes.

Metadata reduction is implemented through complementary controls such as Proxy Network traffic separation, optional VPN Service transport protection, Tor Gateway access for selected web surfaces, Privacy-Preserving Device Handles, traffic shaping where used, and retention limits. These controls reduce exposure and lower confidence in simple communication-pattern inference; they are not documented as complete traffic-analysis resistance.

## Trust boundaries

The primary trust boundaries are:

- User to Enigm
- Enigm to identity platform
- Enigm to secure device management
- Enigm to optional Enigm OS posture
- Enigm to VPN Service and Proxy Network
- Enigm Command to Enigm Server management
- Enigm Command to Tor Gateway access for selected web surfaces
- Enigm to Enigm Intelligence outcomes
- Enigm Command to administrative policy
- OTA Architecture to device verification

## Threat model references

Relevant threat-model areas include account and app compromise, device lifecycle abuse, Enigm OS policy bypass, network-policy misuse, intelligence manipulation, OTA integrity failure, Enigm Command abuse, and loss of audit visibility.

## Failure behavior

The platform should deny protected operations when required identity, authorization, device posture, privacy-preserving device handle, policy, OTA integrity, Remote Attestation outcome, or intelligence state cannot be established.
